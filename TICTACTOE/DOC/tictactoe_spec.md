# 三目並べ（Tic-Tac-Toe）仕様書

<a id="sec-1"></a>
## 1. 概要
コンソール上で動作する三目並べゲームをC言語で実装する。
以下の3つの対戦モードをサポートする。

- **2人対戦モード**：人間同士が交互に手番を行う。
- **CPU対戦モード**：人間1人とコンピュータ（CPU）が対戦する。
- **CPU同士対戦モード（観戦モード）**：CPU同士が対戦する様子を人間が観戦する。

ゲームの状態（盤面・手番・モードなど）は `GameState` 構造体にまとめて管理し、
各関数はこの構造体へのポインタを引数として受け取って操作する
（詳細は[6章](#sec-6)、関数一覧は[10章](#sec-10)、構造の詳細は[UML仕様書](tictactoe_uml.md)を参照）。

<a id="sec-2"></a>
## 2. ゲームルール
- 盤面は3×3のマス目とする。
- 記号は `X` と `O` の2種類。先手は常に `X`。
- プレイヤーは交互に手番を行い、空いているマスに自分の記号を配置する。
- 縦・横・斜めのいずれかの列に同じ記号が3つ揃った時点で、その記号のプレイヤーの勝利とする。
- 全マスが埋まる前でも、これ以上誰も勝てないことが確定した時点で「引き分け」とする（[7.1節](#sec-7-1)）。

<a id="sec-3"></a>
## 3. 対戦モード仕様

<a id="sec-3-1"></a>
### 3.1 モード選択
ゲーム開始時に以下を選択させる。

```
=== 三目並べを開始します ===
対戦モードを選んでください。
1: 2人対戦
2: コンピュータと対戦
3: コンピュータ同士の対戦（観戦モード）
選択(1-3):
```

- 有効な1文字（`1`〜`3`）以外はエラーとし、再入力を促す（入力方式の詳細は[5章](#sec-5)）。

<a id="sec-3-2"></a>
### 3.2 先手・後手の選択（CPU対戦モードのみ）
モード2（CPU対戦）を選んだ場合のみ、人間側が先手(`X`)・後手(`O`)のどちらを担当するかを選択させる。

```
先手(X)・後手(O)どちらで対戦しますか？
1: 先手 (X)
2: 後手 (O)
選択(1-2):
```

モード3（CPU同士対戦）では、この選択は行わない（両方ともCPUが担当するため）。

<a id="sec-3-3"></a>
### 3.3 CPUの思考ルーチン
CPUの手はミニマックス法（Minimax Algorithm）によって決定する。任意の記号（`X` または `O`）を
指定して、その記号にとっての最善手を計算できるように一般化する。

- 盤面のすべての空きマスに対して仮に手を打ち、その後の展開を再帰的に評価する。
- 評価値は「指定記号の勝ち：+10-手数」「相手の勝ち：手数-10」「引き分け：0」とし、
  手数が浅いほど（早く決着がつくほど）評価値の絶対値を大きくすることで、
  「勝てるときは最短で勝ち、負けるときは最長まで粘る」手を選択する。
- 評価値が最大となる手を選択する。
- モード2・モード3ともに同じ思考ルーチンを共用する（モード3では手番ごとに手番側の記号を指定して呼び出す）。
- CPU同士対戦では、両者が最善手を打ち合うため、対局は理論上「引き分け」に終わる
  （どちらかが定石を外さない限り、先勝ちすることはない）。
- CPUが着手した場合、盤面表示の後に「CPU（O）はマス5を選びました。」のように選んだマスを表示する
  （表示順序の理由は[4.2節](#sec-4-2)を参照）。

<a id="sec-3-4"></a>
### 3.4 観戦モードの進行制御（モード3のみ）
- CPU同士の対戦は一瞬で終わってしまうため、1手ごとに盤面を表示した後、
  「(何かキーを押すと次へ進みます)」と表示してユーザーの入力を待つ。
- ユーザーが何らかの1文字を入力すると、その場で（Enterキーを押さなくても）次の手番の処理に進む
  （実現方法は[5章](#sec-5)を参照）。
- 入力ストリームがEOFになった場合はゲームを終了する。

<a id="sec-4"></a>
## 4. 画面表示仕様

<a id="sec-4-1"></a>
### 4.1 盤面表示
盤面は以下のような形式で表示する。空きマスには1〜9の数字を表示する。

```
 1 | 2 | 3
-----------
 4 | 5 | 6
-----------
 7 | 8 | 9
```

手が進むと、該当マスに `X` または `O` が表示される。

**盤面を表示する直前に、画面をクリアする。**
画面クリアにはANSIエスケープシーケンス（`\033[2J\033[H`：画面全体の消去とカーソルを
左上へ戻す）を用いる。これにより、手番が進むたびに画面が更新され、常に最新の盤面だけが
見やすく表示される。

<a id="sec-4-2"></a>
### 4.2 メッセージ表示
- ゲーム開始時：対戦モード選択の案内を表示。
- 手番ごとに現在の手番プレイヤーの記号を表示。
- 人間の手番：「プレイヤー X の番です。マス番号(1-9)を入力してください:」
- CPUの手番（モード2・モード3共通）：着手後、盤面表示の**後に**「CPU（O）はマス5を選びました。」を表示する。
  - 盤面表示の直前に画面クリアが入る仕様（[4.1節](#sec-4-1)）のため、着手メッセージを盤面表示より先に出すと
    直後のクリアで即座に消えてしまう。そのため着手メッセージは、消されない位置（盤面表示の直後）に表示する。
- モード3では、盤面表示のたびに「(何かキーを押すと次へ進みます)」を表示して一時停止する。
- 不正な入力にはエラーメッセージを表示し、再入力を促す。
- 勝敗が決した場合のメッセージはモードにより異なる：
  - モード1：「プレイヤー X の勝ちです！」
  - モード2：「あなた（X）の勝ちです！」または「コンピュータ（O）の勝ちです！」
  - モード3：「CPU（X）の勝ちです！」
- 引き分けの場合：
  - 全マスが埋まっていた場合：「引き分けです。」
  - 空きマスを残して確定した場合：「空きマスは残っていますが、これ以上誰も勝てないため引き分けです。」

<a id="sec-5"></a>
## 5. 入力仕様

<a id="sec-5-1"></a>
### 5.1 入力方式：Enterキー不要の1文字入力
本プログラムでは `scanf` は使用せず、`read()`（POSIXシステムコール）と `termios` を用いて、
Enterキーを押さなくても有効な1文字を入力した瞬間に処理が進むようにする。

- 標準入力がTTY（対話的な端末）に接続されている場合、`termios` によって
  端末の行バッファリング（`ICANON`）とエコー（`ECHO`）を一時的に無効化し、
  1バイト読み取り次第すぐに `read()` が復帰するようにする
  （`c_cc[VMIN]=1`, `c_cc[VTIME]=0` を明示的に設定する。詳細は[UML仕様書](tictactoe_uml.md)のクラス図・実装コメントを参照）。
- 読み取った文字はエコーが無効化されているため、プログラム側で手動で画面に表示する。
- 標準入力がTTYでない場合（パイプ・リダイレクト等）は、通常通り1バイトずつ読み取る。
- 改行文字（`\n`, `\r`）は読み飛ばし、意味のある最初の1文字だけを有効な入力として扱う。
- 入力待ちの直前には、出力バッファを明示的にフラッシュする（`fflush(stdout)`）。
  `read()` を直接使う方式では、stdioが提供する「入力待ち前に出力を自動フラッシュする」機構が
  働かないため、これを怠るとプロンプトが画面に表示されないまま入力待ちになってしまう。
- 入力ストリームがEOFになった場合は、安全にゲームを終了する（無限ループ防止）。

<a id="sec-5-2"></a>
### 5.2 入力チェック項目
- 人間プレイヤーはマス番号(1-9)を1文字入力し、対応するマスに自分の記号を置く。
- 入力チェック項目：
  - 1〜9以外の文字
  - 既に記号が置かれているマスへの入力
- 不正入力の場合はエラーメッセージを表示し、同じプレイヤーに再入力させる（手番は消費しない）。
- CPUの手番では人間側の入力受付は行わない（CPUが自動的に手を決定する）。
- モード3では、着手用の入力は一切発生せず、進行制御用の「何かキー待ち」のみが発生する。

<a id="sec-6"></a>
## 6. 内部データ仕様
ゲームの状態は、以下のフィールドを持つ `GameState` 構造体にまとめて管理する
（グローバル変数は使用しない）。

| フィールド | 型 | 役割 |
|---|---|---|
| `board` | `char[9]` | 盤面。空きマスは `'1'`〜`'9'`、埋まっていれば `'X'` か `'O'` |
| `current_player` | `char` | 現在の手番の記号 |
| `game_mode` | `int` | 1: 2人対戦, 2: CPU対戦, 3: CPU同士対戦 |
| `human_mark` | `char` | モード2時の人間側の記号（モード1・3では未使用） |
| `cpu_mark` | `char` | モード2時のCPU側の記号（モード1・3では未使用） |

`main()` はこの構造体の実体を1つだけ持ち、以降の各関数にはこの構造体へのポインタ
（読み取りのみの場合は `const GameState *`）を引数として渡す。

<a id="sec-7"></a>
## 7. 勝敗判定仕様
以下8パターンのいずれかで同一記号が揃った場合、勝利と判定する。

- 横方向：3パターン（1-2-3, 4-5-6, 7-8-9）
- 縦方向：3パターン（1-4-7, 2-5-8, 3-6-9）
- 斜め方向：2パターン（1-5-9, 3-5-7）

<a id="sec-7-1"></a>
### 7.1 引き分け判定の早期化
引き分けは、以下のいずれかの条件を満たした時点で判定する（全マスが埋まるのを待たない）。

1. **全マスが埋まった場合**（従来通り）。
2. **空きマスが残っていても、8パターンの勝ち筋すべてに `X` と `O` の両方が
   既に含まれてしまい、どちらの記号ももう3つ揃えられない場合**。
   - 例：ある勝ち筋（例：1-2-3）に既に `X` と `O` が1つずつ入っていれば、
     残りのマスが何であってもその勝ち筋は完成しない。
   - この判定を8パターン全てについて行い、すべてが「完成不可能」であれば、
     残りマス数によらずその時点で引き分けを確定させる。
- 全マス消化前に確定した引き分けと、全マス消化後に確定した引き分けとで、表示メッセージを変える
  （[4.2節](#sec-4-2)参照）。

<a id="sec-8"></a>
## 8. 画面クリア仕様
- 盤面を表示する関数は、表示処理の先頭で必ず画面クリアを行う。
- 画面クリアの実現方法：ANSIエスケープシーケンス `\033[2J\033[H` を出力する。
  - `\033[2J`：画面全体を消去する。
  - `\033[H`：カーソルを左上（1行1列目）に移動する。
- 画面クリアに対応していない端末では、エスケープシーケンスがそのまま文字として
  表示されてしまう可能性がある（動作環境は[11章](#sec-11)を参照）。

<a id="sec-9"></a>
## 9. 処理フロー

```
1. 対戦モード選択（2人対戦 / CPU対戦 / CPU同士対戦）
2. （モード2の場合）人間の先手・後手選択
3. 盤面初期化
4. 画面クリア＋盤面表示
5. 繰り返し：
   5.1 現在の手番を判定
       - モード1: 常に人間が入力
       - モード2: 手番がCPU側なら自動着手、人間側なら入力受付
       - モード3: 常にCPU（現在の手番側の記号）が自動着手
   5.2 盤面更新
   5.3 画面クリア＋盤面表示
   5.4 （CPUの手番だった場合）着手したマスをメッセージ表示
   5.5 （モード3の場合）何かキーが押されるまで待機
   5.6 勝敗判定
       - 勝者あり → 結果表示して終了
       - 引き分け（[7.1節](#sec-7-1)の条件を満たす）→ 結果表示して終了
   5.7 プレイヤー交代
6. 「もう一度遊びますか？」を確認
```

## 10. 使用関数（想定）
| 関数名 | 役割 |
|---|---|
| `getGameMode()` | 対戦モード（1〜3）の選択・入力受付 |
| `getHumanMark()` | CPU対戦モード（モード2）時の先手・後手選択 |
| `readRawByte()` | termiosによるrawモードで1バイトを即時読み取る（低レベル入力） |
| `readMenuChar()` | `readRawByte()`を用い、改行を読み飛ばして意味のある1文字を返す |
| `waitForEnter()` | モード3における、何かキーが押されるまでの一時停止 |
| `clearScreen()` | ANSIエスケープシーケンスによる画面クリア |
| `initBoard(game)` | 盤面の初期化 |
| `printBoard(game)` | 画面クリア＋盤面の表示 |
| `isEmptyCell(game, index)` | 指定マスが空きかどうかの判定 |
| `getMove(game)` | 人間プレイヤーからの入力受付とバリデーション |
| `getOpponentMark(mark)` | 指定した記号の相手側の記号を返す |
| `getBestMove(game, forMark)` | ミニマックス法により指定記号にとっての最善手を算出（モード2・3共用） |
| `minimax(game, forMark, depth, isMaximizing)` | 再帰的な盤面評価（ミニマックス法の本体） |
| `updateBoard(game, index)` | 盤面の更新 |
| `checkWin(game)` | 勝敗判定（勝者の記号 or 0 を返す） |
| `isBoardFull(game)` | 全マスが埋まっているかどうかの判定 |
| `isNoWinPossible(game)` | 全ての勝ち筋にX・Oの両方が含まれ、これ以上勝者が出ないかどうかの判定 |
| `checkDraw(game)` | 引き分け判定（`isBoardFull()` または `isNoWinPossible()` が真なら引き分け） |
| `switchPlayer(game)` | 手番の交代 |
| `isComputerTurn(game)` | 現在の手番がCPUかどうかをモードに応じて判定 |
| `announceWinner(game, winner)` | モードに応じた勝敗メッセージの表示 |
| `announceDraw(game)` | 全マス消化前/後で異なる引き分けメッセージを表示 |
| `playOneTurn(game)` | 1手番分の処理（着手→更新→表示→一時停止） |
| `playOneGame(game)` | 1ゲームを最初から決着まで実行 |
| `askPlayAgain()` | 「もう一度遊びますか？」の確認 |
| `main()` | 全体の制御フロー（`GameState`の実体を保持） |

<a id="sec-11"></a>
## 11. 拡張候補（任意）
- CPUの強さ（難易度）を選べるようにする（例：ランダム手を混ぜた「弱いCPU」）
- CPU同士対戦で、片方または両方にわざと最善手を外させて「先勝ち」する展開も見られるようにする
- 対戦成績の記録
- 盤面サイズの可変化（N目並べ）
- Windows環境向けに `conio.h` の `_getch()` を使った入力方式を `#ifdef _WIN32` で切り替えられるようにする

<a id="sec-12"></a>
## 12. 動作環境
- 言語：C（C99以降を想定）
- 実行環境：POSIX準拠のコンソール環境（Linux / macOS のターミナル）
  - `termios.h` を使用しているため、Windowsの標準コンソールでは**そのままでは動作しない**
    （別途 `conio.h` 等を使った実装が必要）。
  - 画面クリアにANSIエスケープシーケンスを使用しているため、これに対応した端末が前提となる。
- 特別な外部ライブラリは使用しない（標準ライブラリ＋POSIX標準関数のみ）。
