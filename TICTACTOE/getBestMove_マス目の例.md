# `getBestMove` の動作説明（マス目の例つき）

`getBestMove_詳細解説.md` / `minimax_詳細解説.md` の内容を踏まえ、
実際のマス目を使った具体例で`getBestMove`の動きを追う。

---

## 例1：CPUが「今すぐ勝てる」局面

次の盤面を考える（`forMark = 'O'`、CPUの番）。

```
 X | 2 | 3
-----------
 O | O | 6
-----------
 X | 8 | 9
```

- 埋まっているマス：1(`X`), 4(`O`), 5(`O`), 7(`X`)
- 空いているマス：2, 3, 6, 8, 9

`board`配列で見ると、`board[3]='O'`（マス4）、`board[4]='O'`（マス5）がすでに揃っており、
**マス6（`board[5]`）に`O`を置けば横一列（4-5-6）が完成して即勝利**、という状況である。

### `getBestMove(game, 'O')`の動き

```c
for (i = 0; i < 9; i++) {
    if (isEmptyCell(game, i)) {
        ...
        int score = minimax(game, 'O', 0, 0);
        ...
    }
}
```

`i`は`0`〜`8`の順に回るので、空いているマスの順番（インデックス）は
`1`(マス2) → `2`(マス3) → `5`(マス6) → `7`(マス8) → `8`(マス9) となる。

| i (インデックス) | マス番号 | 仮に`O`を置いた結果 | `minimax`が返す`score` | `bestScore`/`bestMove`の更新 |
|---|---|---|---|---|
| 1 | 2 | まだ決着せず、この先を深く先読み | 例：`-1`くらい（相手にも勝ち筋が残るため） | `bestScore=-1`, `bestMove=1` |
| 2 | 3 | 同様にまだ決着せず | 例：`0`くらい | `0 > -1` → `bestScore=0`, `bestMove=2` |
| **5** | **6** | **横一列(4-5-6)が`O,O,O`で即勝利！** | **`10`**（`checkWin`が`forMark`の勝ちを検出し`10 - depth`＝`10 - 0`） | `10 > 0` → `bestScore=10`, `bestMove=5` |
| 7 | 8 | まだ決着せず、深く先読み | `10`を超えることはあり得ない（評価値の理論上の最大が10のため） | 更新されない |
| 8 | 9 | 同上 | 同上 | 更新されない |

### なぜ`i=5`（マス6）で`score=10`が返るのか

`getBestMove`が`board[5]`に仮に`'O'`を置いた**直後**に`minimax(game, 'O', 0, 0)`を呼ぶ。
`minimax`は関数の先頭でまず`checkWin(game)`を呼ぶが、この時点で盤面はすでに
「4-5-6が`O,O,O`」という**勝敗が決した状態**になっている。そのため、1回もループを
回すことなく、一番最初の

```c
if (winner == forMark) return 10 - depth;   // ここで即座にreturn
```

がヒットし、`depth`はまだ`0`なので`10 - 0 = 10`がそのまま返る。

### 結果

`bestScore=10`が理論上の最大値であり、他のどのマスを試してもこれを上回ることはできないため、
最終的に`getBestMove`は`bestMove = 5`（マス番号でいうと`6`）を返す。
CPUはこの手を選び、「CPUはマス 6 を選びました。」と表示されて勝利する。

## 例2：即勝ちではなく「相手を防ぐ」局面

たとえば逆に、盤面が

```
 X | X | 3
-----------
 O | 5 | 6
-----------
 7 | 8 | 9
```

のようにマス1・2に`X`が並んでいて、CPU(`O`)がマス3を塞がないと次にXが勝ってしまう、
という局面だとする。この場合、マス3（`board[2]`）に`O`を仮置きしても
その場ではまだ誰も勝っていないため`score`は`10`のような即決値にはならない。
しかし`minimax`が「マス3を塞がなかった場合」の枝を再帰的に評価すると、
その先で必ずXが勝ってしまう（＝低い評価値になる）ため、
**相対的に見てマス3を塞ぐ手が一番評価値が高くなり**、結果として選ばれる。

「即座に高得点が出るから選ばれる」のではなく、「他の手を選ぶと将来的にもっと
悪い結果になると`minimax`が見抜くから、消去法的に選ばれる」という点が、
例1（即勝ち）との違いである。

---

## まとめ

| 観点 | 例1：即勝ちできる局面 | 例2：相手を防ぐ必要がある局面 |
|---|---|---|
| 仮置き直後の`checkWin`の結果 | その場で`forMark`の勝ちが確定する | まだ決着しない |
| `minimax`が値を返すタイミング | 再帰の1階層目（`depth=0`）で即`return` | さらに先まで再帰した末に、間接的に低い/高い評価値が伝播してくる |
| 評価値が高くなる理由 | 「勝ったから」直接的に`10 - depth`が返る | 「この手を選ばないと将来的に負けるから」相対的に高くなる |

いずれの場合も、`getBestMove`のループ自体がやっていることは同じで、
「9マスのうち空いている各マスに仮に置いてみて、`minimax`が返す評価値を比較し、
一番良かったマスを選ぶ」という単純な処理である。局面によって`minimax`が
即座に決着を検出するか、何手も先まで読んでから間接的に評価するかが異なるだけである。
