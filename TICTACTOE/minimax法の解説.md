# ミニマックス法（Minimax Algorithm）の解説

このドキュメントは、`tictactoe.c`の実装コードそのものの解説（`minimax_詳細解説.md`）とは別に、
**ミニマックス法という考え方自体**を、三目並べを例にして解説するものである。

---

## 1. ミニマックス法とは

ミニマックス法は、**2人が交互に手を打ち、片方が得すればもう片方が同じだけ損をする
（ゼロサム）ゲーム**において、「相手が常に最善の手を打ってくる」と仮定した上で、
自分にとって最も良い結果になる手を選ぶための探索アルゴリズムである。

三目並べ、チェス、将棋、オセロなど、多くの2人対戦ボードゲームのAIの基礎になっている。

- 名前の由来：自分の番では評価値を**最大化(Max)**しようとし、相手の番では
  相手が自分にとっての評価値を**最小化(Min)**しようとする、という2つの立場を
  交互に想定することから「ミニ・マックス」と呼ばれる。

## 2. 基本的な考え方

```
自分の番（最大化したい）
  └─ 相手の番（最小化したい）
        └─ 自分の番（最大化したい）
              └─ 相手の番（最小化したい）
                    └─ ...（決着がつくまで繰り返す）
```

- **自分の番**：これから打てる手を全部試し、その中で一番評価値が高くなる手を選ぶ
  （＝自分に一番都合が良い未来を選ぶ）。
- **相手の番**：相手が打てる手を全部試すが、相手は自分（評価値を計算している側）
  にとって一番都合が悪い手を選んでくると仮定し、その中で一番評価値が低くなる手を
  採用する（＝相手が最善を尽くした場合の最悪の想定をする）。

この「自分の番なら最大、相手の番なら最小」を、盤面が決着するまで**交互に繰り返し
ながら再帰的に**評価する。決着がついた局面（勝ち・負け・引き分け）まで完全に読み切って
初めて数値が確定し、その数値が1手ずつ手前に伝播していく。

## 3. なぜ「相手も最善を尽くす」と仮定するのか

もし「相手はミスをするかもしれない」という期待を組み込んでしまうと、相手が実際に
ミスをしなかった場合に不利な手を選んでしまうリスクがある。ミニマックス法は
「相手は絶対に最善を尽くしてくる」という最も厳しい前提を置くことで、
**どんな相手が来ても、少なくともこれ以上悪い結果にはならない**手を選べる、
という保証（ゲーム理論でいう「最悪の場合を想定した最適戦略」）を得ている。

## 4. 三目並べでの評価値の設計

`tictactoe.c`の実装では、評価値を次のように設計している。

| 局面 | 評価値 |
|---|---|
| 評価対象（`forMark`）の勝ち | `10 - depth` |
| 相手の勝ち | `depth - 10` |
| 引き分け | `0` |

`depth`は「決着までに何手かかったか」を表す。これを評価値に組み込むことで、

- 勝てるなら**できるだけ早く**勝つ手を選ぶ（`depth`が小さいほど`10 - depth`が大きい）。
- 負けが避けられないなら**できるだけ長く粘る**手を選ぶ（`depth`が大きいほど
  `depth - 10`が0に近づき、マシになる）。

という優先順位が自然に生まれる。単に「勝ち=+10、負け=-10、引き分け=0」とだけ決めて
`depth`を無視すると、1手で勝てる場合も5手かけて勝つ場合も同じ評価値になってしまい、
無駄に長引かせる手を選んでしまうことがある。

## 5. 具体例で見る「探索木」

以前取り上げた、次の局面（Oの番）を例にする。

```
 X | O | 3
-----------
 X | X | 6
-----------
 O | 8 | 9
```

Xはすでに「4-5-6」と「1-5-9（対角線）」の**2つの勝ち筋**を同時に狙える状態
（いわゆる「フォーク」）になっている。この場合の探索木を簡略化すると：

```
Oの番（今の局面）
├─ Oがマス3に打つ
│    └─ Xの番：マス6 か マス9 のどちらかに打てば即座にXの勝ち → 評価値 -9
├─ Oがマス6に打つ（4-5-6の脅威を塞ぐ）
│    └─ Xの番：マス9に打てば対角線が揃いXの勝ち → 評価値 -9
├─ Oがマス8に打つ
│    └─ Xの番：マス6 か マス9 のどちらかに打てば即座にXの勝ち → 評価値 -9
└─ Oがマス9に打つ（対角線の脅威を塞ぐ）
     └─ Xの番：マス6に打てば4-5-6が揃いXの勝ち → 評価値 -9
```

Oがどのマスに打っても、**1手先でXが必ず勝てる**（評価値`-9` = `depth(1) - 10`）ため、
4つの候補手すべてが同じ評価値になる。ミニマックス法はこの時点で「もう手遅れ
（Oの敗着はもっと前の手にあった）」ことを数値として検出できる。
（実際にこの局面を`tictactoe.c`のロジックでシミュレートした結果とも一致する。）

このように、ミニマックス法は「勝てる手」だけでなく「詰んでいることの検出」も
同じ枠組みで自然に行える点が特徴である。

## 6. なぜ三目並べでは「CPUが絶対に負けない」のか

三目並べは盤面が3×3の9マスしかなく、**とりうる局面の総数が非常に少ない**
（理論上の上限は9!=362,880通りだが、決着したらそこで打ち切られるため実際はもっと少ない）。
ミニマックス法はこの全パターンを漏れなく読み切れるため、**「解析され尽くしたゲーム」**
になる。相手が最善を尽くしても、こちらも同じく最善を尽くせば理論上の最良の結果
（勝てるなら勝ち、無理なら引き分け）を必ず得られる。

これがチェスや将棋のように盤面のパターンが天文学的な数になるゲームだと、
決着まですべて読み切ることは現実的な時間内には終わらない。そのため、
実際のチェスAIなどでは、

- **アルファベータ枝刈り**：明らかに選ばれない枝を計算する前に切り捨てる高速化。
- **深さ制限＋評価関数**：決着まで読み切らず、途中の局面を「形勢の良し悪し」を
  推定する関数（評価関数）でスコア化して打ち切る。

といった工夫が必須になる（`tictactoe.c`の規模では不要なため、いずれも実装していない）。

## 7. `tictactoe.c`における実装との対応

| ミニマックス法の考え方 | 対応する実装 |
|---|---|
| 自分の番／相手の番の切り替え | `isMaximizing`引数（`1`=自分の番、`0`=相手の番） |
| 決着までの深さ | `depth`引数 |
| 「自分の番なら最大化」 | `bestScore`を`score > bestScore`で更新（大きい方を採用） |
| 「相手の番なら最小化」 | `bestScore`を`score < bestScore`で更新（小さい方を採用） |
| 決着判定（再帰の終了条件） | `checkWin()` / `checkDraw()` |
| 「勝てるなら早く、負けるなら粘る」 | 評価値に`depth`を組み込む（`10 - depth` / `depth - 10`） |
| 実際に手を決める入口 | `getBestMove()`（探索木の1段目だけを担当し、以降は`minimax()`に委譲） |

コードレベルの詳細（変数・呼び出し関係など）は`minimax_詳細解説.md`、
`getBestMove`単体の詳細は`getBestMove_詳細解説.md`を参照。
